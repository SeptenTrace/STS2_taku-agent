# STS2 Taku Agent 总体规划

## 1. 项目目标

这个 mod 的长期目标不是简单地做一个自动脚本，而是构建一个可以:

- 读取《杀戮尖塔 2》当前对局状态
- 基于规则与上下文做出动作决策
- 在地图、资源、战斗节奏之间进行长期规划

为了降低复杂度，整个项目拆成三个阶段推进。每个阶段都要有独立可验收的产物，避免一开始就把“读状态、出牌、规划”混在一起。

## 2. 三阶段拆分

## 阶段一：观测层 / State Observation

目标是让 AI 可以稳定、结构化地“看懂”游戏。

这个阶段不追求让 AI 真正打牌，重点是建立一套可靠的数据提取能力，让外部策略层可以读取:

- 当前战斗状态
- 玩家状态
- 敌人状态
- 手牌、抽牌堆、弃牌堆、消耗区
- 药水、遗物、姿态、buff / debuff
- 回合、能量、格挡、意图、伤害预估等关键字段

核心交付物:

- 一套统一的游戏状态模型
- 一套从游戏运行时抓取状态的接口
- 一套调试/导出工具，用于验证读取结果是否正确
- 一套最小可用的快照输出机制，供后续 AI 决策模块消费

验收标准:

- 在一场普通战斗中，能够稳定导出每回合状态快照
- 快照中的玩家、敌人、卡牌区、药水、遗物信息与 UI 基本一致
- 在常见战斗场景下，不因为动画、过渡状态或事件回调顺序导致数据严重错乱

## 阶段二：动作层 / Action Execution

目标是让 AI 可以对游戏施加有效动作。

这个阶段要解决的不是“AI 聪不聪明”，而是“AI 能不能可靠操作游戏”。动作接口需要覆盖:

- 打出一张合法手牌
- 选择目标
- 处理需要二次交互的牌或效果
- 使用药水
- 结束回合
- 在事件、奖励、选项界面中完成基础选择

核心交付物:

- 一套动作抽象层，把游戏操作统一成有限动作空间
- 一套动作执行器，把 AI 决策映射成游戏内操作
- 一套合法性检查，避免执行非法动作
- 一套失败恢复机制，处理动作超时、界面切换、目标失效等情况

验收标准:

- AI 可以在一个标准战斗中独立完成多个回合
- 可以打出普通攻击牌、防御牌和至少一类需要目标选择的牌
- 可以识别“当前无可执行动作”并正确结束回合
- 动作日志与游戏实际结果可对应

## 阶段三：长期规划层 / Strategic Planning

目标是让 AI 不只是“会操作”，而是“会推进一局游戏”。

这个阶段关注跨回合、跨战斗、跨楼层的策略能力，包括:

- 阅读地图节点
- 规划路线
- 评估精英、商店、篝火、事件收益
- 在战斗中基于后续风险调整资源使用
- 在卡牌奖励、遗物选择、升级、移除卡牌时做长期决策

核心交付物:

- 一套全局 Run State 模型
- 一套路线路径评估器
- 一套奖励选择与资源规划策略接口
- 一套“战斗内短期策略”和“战斗外长期策略”解耦的架构

验收标准:

- AI 可以独立完成地图节点选择
- 能在多个可行路线中给出明确偏好
- 战斗决策能受长期资源约束影响，而不是只看当前回合
- 可以完整推进一局 run 的大部分流程

## 3. 推荐的技术分层

为了避免后面返工，建议从一开始就把 mod 拆成四层:

### 3.1 Runtime Hook Layer

负责接入游戏对象、生命周期和事件回调。

职责:

- 找到游戏内可读取的对象与字段
- 在合适时机采样状态
- 在合适时机执行动作

### 3.2 Domain Model Layer

负责把游戏内复杂对象转换成稳定、可序列化的数据模型。

职责:

- 定义 `BattleState`、`PlayerState`、`EnemyState`、`CardState` 等结构
- 对游戏内部对象做裁剪和标准化
- 尽量避免把 AI 逻辑直接绑定到游戏原始类

### 3.3 AI Interface Layer

负责向外暴露统一接口。

职责:

- 提供状态快照
- 接收动作请求
- 输出日志、调试信息和错误信息

### 3.4 Strategy Layer

负责具体决策逻辑，可以先用规则系统，之后再接 LLM、搜索、强化学习或混合方案。

职责:

- 根据状态快照选择动作
- 根据地图与奖励做长期规划
- 保持与底层执行层解耦

## 4. 依赖关系

三个阶段严格依赖:

1. 阶段一先解决“看得见”
2. 阶段二再解决“动得了”
3. 阶段三最后解决“想得远”

如果阶段一的数据模型不稳定，阶段二和阶段三会持续返工。这个项目的关键不在于尽快接入模型，而在于尽快建立可靠的状态与动作边界。

## 5. 每阶段建议里程碑

### M1: 基础状态快照可导出

- 能输出玩家、敌人、手牌、能量、格挡、意图

### M2: 完整战斗状态可导出

- 增加遗物、药水、buff/debuff、牌堆信息、回合阶段

### M3: 最小动作闭环

- 能打牌、选目标、结束回合

### M4: 单场战斗自动游玩

- 能在简单战斗中连续执行直到结束

### M5: 地图与奖励策略接入

- 能选择路径、奖励、休息点行为

## 6. 主要风险

- 游戏对象生命周期和 UI 状态未必同步，容易读到半更新状态
- 某些牌或效果需要多步交互，动作建模会比普通出牌复杂
- 状态字段很多，前期如果没有统一快照模型，后面会难以维护
- 长期规划如果过早开始，会被底层观测和执行不稳定拖死

## 7. 当前建议

阶段一目前已经完成，因此当前建议切换为“在不破坏观察契约的前提下推进阶段二”。

当前策略是:

- 保持 `CLI + server + layered observation` 不变
- 阶段二动作接口直接依赖 `context / actions / delta observation`
- 先打通战斗执行闭环，再扩展局外动作
- 不要在阶段二里回退成全量状态驱动，继续保持低 token 查询路径
