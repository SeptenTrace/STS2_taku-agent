# STS2 Taku Agent 可行性验证

## 1. 目标

这个目录只做一件事:

按照项目已经定义的三个阶段，验证这些目标在当前游戏架构下是否可行。

这里的验证原则是:

- 优先使用公开的 mod 入口、公开类型和运行时对象
- 优先验证“是否能读 / 是否能调 / 是否能稳定复现”
- 只有在公开入口无法提供足够信息时，才考虑定向反编译
- 只做 mod 开发，不把游戏反编译代码或还原源码提交到仓库

## 2. 当前方法边界

目前的可行性判断基于:

- 本地游戏程序集元数据与类型结构
- 已能确认的命名空间、公开类型、公开方法和公开属性
- 现有 mod 初始化能力

目前不依赖:

- 大规模反编译后的游戏源码整理
- 将游戏代码复制进仓库
- 修改游戏原始文件

## 3. 当前总体结论

### 阶段一：高可行

理由:

- 已确认存在战斗状态、Hook、玩家回合、卡牌出牌、药水使用等相关公开入口
- 已确认 `CombatState` 暴露敌我单位、回合侧、遭遇对象等核心信息
- 已确认 `Hook` 提供大量战斗生命周期回调，足够支撑快照采样

### 阶段二：中高可行

理由:

- 已确认存在 `CardCmd`、`PlayerCmd`、`CardSelectCmd`、`PotionCmd` 等命令入口
- 已确认存在结束回合、自动打牌、选牌等公开方法
- 复杂交互是否能完全覆盖，还需要逐个场景验证

### 阶段三：中等可行

理由:

- 已确认存在地图、奖励、休息点、事件、遗物、商店相关命名空间和类型
- 已确认存在 `AutoSlay` 相关自动处理器，说明游戏内部有自动推进结构
- 但长期规划涉及地图、奖励、战斗外界面、跨楼层状态，需要更多运行时验证

## 4. 验证顺序

建议按下面顺序推进，不跳步:

1. 阶段一先验证战斗状态能否稳定读取
2. 阶段二再验证动作接口能否稳定执行
3. 阶段三最后验证地图、奖励与长程决策入口

如果阶段一未打稳，不进入阶段二。
如果阶段二动作闭环不稳定，不进入阶段三。

## 5. 反编译使用原则

反编译不是默认路线，只在下面场景使用:

- 公开类型名已知，但字段路径不清楚
- 公开方法存在，但调用前置条件不清楚
- 运行时能拿到对象，但不知道关键属性挂在哪一层
- 某些复杂交互仅靠公开签名无法判断执行方式

使用方式:

- 只做定向反编译
- 只查当前阶段卡点相关类型
- 只在本地分析
- 只把结论、字段名、调用关系整理成我们自己的文档

明确禁止:

- 把反编译得到的游戏源码提交到本仓库
- 大段复制游戏实现代码到 mod 中
